# BA 系統 - 授權處理：離線版與線上版

本文件說明**系統授權（授權碼解鎖模組）的兩種實作方式：離線版與線上版**。可擇一實作，或採混合（例如離線驗證為主、線上用於報到／延長）。  
與登入、角色權限的關係見 `docs/00-decisions/license-feature-gate.md` 與 `docs/40-contracts/api-surface.md`。

---

## 共通前提

- **授權維度**：以 Feature Key 對應模組（如 `core`、`environment`、`people_counting`、`lighting`、`access_control` 等），與前端 `system-modules` 對齊。
- **後端 API**：`GET /api/license`（回傳目前狀態）、`POST /api/license/activate`（admin 輸入授權碼啟用）。
- **檢查順序**：authenticate → 角色權限 → requireFeature。未授權或過期回傳 403，`code: "FEATURE_NOT_LICENSED"`。
- **前端**：登入後取 GET /api/license，以 `features` 過濾導航與頁面；僅 admin 可執行啟用。

以下僅描述「授權碼如何驗證、如何寫入後端」的差異。

---

## 一、離線版（Offline License）

### 概念

- 授權碼是一段**已簽章資料**（例如 JWT 或自訂 binary/base64 格式），內含：可啟用的 features、到期日、選填的客戶/站點識別等。
- 後端僅用**內建公鑰**驗簽，**不需連外網**，適合無法連線或需完全離線部署的場域。

### 授權碼格式（建議）

以 JWT 為例，payload 建議欄位：

| 欄位                 | 說明                                         |
| -------------------- | -------------------------------------------- |
| `features`           | 陣列，如 `["core","environment","lighting"]` |
| `exp`                | 到期時間（Unix timestamp）                   |
| `iat`                | 簽發時間（可選）                             |
| `site` 或 `customer` | 站點／客戶識別（可選，用於綁定或顯示）       |

簽章：後端以**固定公鑰**驗證（私鑰僅在發行授權的環境使用，不部署到客戶端）。

### 流程

1. **發行端（貴司）**：依訂單產生授權碼（含 features、exp），以私鑰簽名，交付客戶（貼紙、郵件、後台下載等）。
2. **客戶端**：admin 在系統內輸入授權碼，前端呼叫 `POST /api/license/activate`，body `{ code: "eyJ..." }`。
3. **後端**：

- 解碼 JWT，以公鑰驗簽；
- 檢查 `exp` 未過期；
- 可選：檢查 `site` 與目前站點是否一致；
- 將通過驗證的 `features` 與 `expiresAt` 寫入儲存（DB 或 settings）；
- 回傳 200 與更新後的 license 狀態。

4. 之後所有授權狀態以**本地儲存**為準，不再依賴網路。

### 優點與限制

| 優點                           | 限制                                                                    |
| ------------------------------ | ----------------------------------------------------------------------- |
| 不需外網，適合封閉環境         | 授權碼一旦流出可被多站重複使用（除非在 payload 綁定 site 並在後端檢查） |
| 實作簡單，僅需密碼學驗簽       | 撤銷需等到期或發新碼覆蓋，無法即時作廢                                  |
| 效能與可用性不受授權伺服器影響 | 需妥善保管私鑰、授權碼發行流程需規範                                    |

### 實作要點（後端）

- 依環境變數或設定讀取**公鑰**（PEM），驗簽使用現成函式庫（如 Node 內建 crypto、jsonwebtoken）。
- 若採用「一機一碼」：payload 可含 `site` / `machineId`，後端比對目前站點再寫入。
- 儲存：與線上版相同，可寫入 DB（如 `license` 表）或 settings（`license_features`、`license_expires_at`）。

## 二、線上版（Online License）

### 概念

- 授權碼為一組**序號／啟用碼**（不內含 features），後端在啟用時**向授權伺服器**請求驗證。
- 授權伺服器回傳：可啟用 features、到期日、可選的綁定資訊（如 site_id）。
- 適合需要**集中控管、即時撤銷、用量統計或延長**的商業模式。

### 授權碼與伺服器角色

- **授權碼**：由授權伺服器發行，可為隨機序號或有意義編碼；**不包含** features 與到期日（這些由伺服器查詢後回傳）。
- **授權伺服器**：貴司自建或委外服務，提供例如 `POST /validate` 或 `GET /license?code=xxx`，回傳 JSON：`{ features: string[], expiresAt: string, ... }`。

### 流程

1. **發行端**：在授權伺服器後台建立訂單／授權，產生授權碼並記錄 features、到期日、可選綁定站點。
2. **客戶端**：admin 輸入授權碼，前端呼叫 `POST /api/license/activate`，body `{ code: "XXXX-XXXX-..." }`。
3. **後端**：

- 呼叫授權伺服器 API（帶 `code`、可選帶目前 site_id）；
- 若伺服器回傳成功：取得 `features`、`expiresAt`，寫入本地儲存；
- 若伺服器回傳無效／過期／已撤銷：回傳 403，body 含 `code: "INVALID_LICENSE"` 等；
- 若網路異常：可回傳 503 或明確錯誤，提示稍後再試。

4. 之後日常運作仍以**本地儲存**的 features 與到期日為準；可選實作**定期與授權伺服器同步**（心跳／重新驗證）以支援即時撤銷或延長。

### 優點與限制

| 優點                             | 限制                       |
| -------------------------------- | -------------------------- |
| 可即時撤銷、延長、統計啟用數     | 依賴網路與授權伺服器可用性 |
| 授權碼可為短碼，不需傳遞大段 JWT | 需建置或整合授權伺服器     |
| 可做綁定（站點、數量）與合規檢查 | 部署環境需能連到授權伺服器 |

### 實作要點（後端）

- 授權伺服器 URL、API Key（若需）等以設定或環境變數配置。
- 呼叫時設定 timeout，並處理連線失敗、4xx/5xx；回傳給前端的錯誤碼與訊息要一致（如 `INVALID_LICENSE`、`LICENSE_SERVER_ERROR`）。
- 本地儲存結構可與離線版相同；若要做「定期同步」，可加 last_validated_at 與同步間隔邏輯。

## 三、比較與選擇

| 項目       | 離線版                         | 線上版                             |
| ---------- | ------------------------------ | ---------------------------------- |
| 網路依賴   | 無（僅啟用時不需連外網）       | 啟用時需連授權伺服器；可選定期同步 |
| 即時撤銷   | 不支援                         | 支援（透過伺服器與同步機制）       |
| 實作複雜度 | 低（驗簽 + 本地儲存）          | 中高（需授權伺服器與網路處理）     |
| 適用情境   | 封閉網路、一次性授權、不需撤銷 | 需集中控管、延長、撤銷、統計       |

可採**混合**：預設離線驗簽啟用，另提供「線上註冊」選項，上傳授權碼到授權伺服器做報到或延長，後端再依伺服器回傳更新本地狀態。
